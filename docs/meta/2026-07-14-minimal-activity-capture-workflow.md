## Минимальная модель взаимодействия с пользователем

Принято решение строить TrailsLog вокруг максимально дешевой фиксации активности.

Основная идея:

Пользователь не должен задумываться о структуре данных во время записи.

Любая лишняя формализация увеличивает вероятность того, что запись вообще не будет сделана.

Поэтому взаимодействие строится следующим образом.

1. Пользователь отправляет команду активности.

2. Активность сразу считается сохраненной.

3. Бот подтверждает сохранение.

4. Пользователь может ответить на сообщение бота.

Ответ может содержать:

- количество;
- единицы измерения;
- комментарий;
- настроение;
- любые дополнительные данные.

На данном этапе эти данные не разбираются.

Они сохраняются как есть.

5. Пользователь может ответить

DEL

или

-

чтобы отменить запись.

Таким образом достигаются две цели.

Во-первых, фиксация остается максимально быстрой.

Во-вторых, появляется возможность постепенно усложнять структуру данных без изменения пользовательского сценария.

На текущем этапе TrailsLog сохраняет сырые данные Telegram.

Интерпретация данных откладывается на более поздние этапы разработки.
